# Android 自动化测试计划设计

## 概述

根据用户提供的测试需求或场景描述，设计结构化的 Android 自动化测试计划。基于 Midscene.js 视觉驱动的特性，计划重点关注 UI 交互、视觉验证和用户行为模拟。

## 步骤

### 1. 需求分析

从用户描述中提取以下信息：
- **被测应用**：App 名称、包名、版本
- **测试场景**：功能模块列表
- **测试环境**：Android 版本、设备型号、网络要求
- **测试目标**：验收测试/回归测试/探索测试
- **特殊约束**：登录态、权限、数据依赖

如有缺失信息，使用 clarify 向用户询问。

### 2. 测试策略设计

根据 Midscene.js 的能力矩阵设计测试策略：

| Midscene 能力 | 适用测试场景 |
|---|---|
| `aiAct()` 自动规划 | 复杂多步骤流程（如注册、下单） |
| `aiTap()` / `aiInput()` 即时操作 | 精确的元素交互验证 |
| `aiAssert()` | UI 状态断言（元素存在/文本匹配） |
| `aiQuery()` | 结构化数据提取与验证 |
| `aiWaitFor()` | 页面加载/状态等待 |
| `aiLocate()` | 元素定位验证 |
| `launch()` / `terminate()` | App 生命周期管理 |
| `runYaml()` | 声明式批量脚本执行 |

### 3. 测试优先级

- **P0（冒烟测试）**：核心路径，必须通过才能继续
- **P1（功能测试）**：主要功能模块验证
- **P2（边界测试）**：异常场景、边界值、兼容性

## 注意事项

- 重点利用 Midscene 的 AI 视觉识别能力设计用例，减少对 UI 元素 ID 的依赖
- 对 Midscene 的 `aiAct` 描述要做到自然语言级别精确
- 考虑 `aiActionContext` 的配置（如弹窗处理策略）
- 如用户未指定模型方案，默认推荐 `qwen3.7-plus`
